数据冒险 
 - 写后读 
 	- 语义不一致: 写后读如果紧挨着执行会导致语义不一致(还没有写入就开始读取操作),所以需要nop进行拖延。
 	- 数据前递: 执行阶段把结果前递给译码阶段(0拖延)，访存阶段把结果前递给译码阶段(1拖延)
 - 读后写
 - 写后写 寄存器重用导致编译器产生写后写和读后写。 寄存器重命名(汇编中的逻辑寄存器(16个)映射到物理寄存器(上百个))用于管理共享资源(避免因寄存器复用导致的假依赖)可以解决写后写和读后写冒险以支持乱序执行
控制冒险
 - 延迟分支
 - 分支预测 利用数据前递在执行阶段确认预测结果，使用flush冲刷错误指令(用nop替换) (retire引退其实是commit)

现代cpu的流水线具有深度流水线(多阶段流水线)和超标量(多流水线)的特点 (指令多发射时可以保证发射顺序，但不保证完成顺序(乱序执行)，所以利用重排缓存进行retire更新架构资源的状态保证完成顺序)
使用重排序缓存和寄存器重命名实现基于硬件的投机执行(同时执行分支的指令，淘汰其一)
寄存器重命名加上数据前递解决了所有数据冒险
现代cpu架构流水线: 取指预解码(识别变长指令，分支预测，多发射) -> 指令队列(容量大，替代指令寄存器) -> 解码和微码(多个解码器并行解码，扩展指令成为多条微操作) -> 寄存器重命名 -> 重排缓存 -> 调度器(把保留站中排队的微操作分配给空闲的功能单元，单周期并行调度) -> 功能单元(高延迟功能会流水线化) -> 引退单元(使用重排缓存进行引退操作)

有效内存访问时间(EMAT) = 在缓存中查找数据所花费的时间(cache access time) +  miss rate * 缓存缺失时从更深层的存储中查找数据的时间(miss penalty)
因为每次访问内存都必然要花费cache access time，而miss penalty只有在miss时才额外产生
流水线中缓存读/写算法
 - 缓存的读访问：读延迟定义为处理一个读缺失所花费的时钟周期数
 - 缓存的写访问：写延迟定义为写操作浪费的处理器时钟周期数(忽略命中或缺失的情况) ???
  - 直写: 直接写缓存和主存，使用store buffer避免访问主存。 写缺失时缓存分配策略:
   - 写分配  写缓存和store buffer
   - 非写分配 不写缓存，只写store buffer
  - 回写: 只写缓存，延后刷回主存。写缺失时缓存分配策略和读缺失一致
  在多级缓存架构中对L1使用直写策略，对L2、L3使用回写策略
缓存写操作需要先加载再写入(写缺失)：主要是由于缓存的数据管理单位是“块”(cache line)而非单个字节或地址。
流水线中缓存缺失处理：不能在取指阶段隐藏缺失，但可以在访存阶段隐藏缺失
 - 访存阶段读缺失：数据前递 or 指令重排
 - 访存阶段写缺失：直写+写分配 和读缺失一致；直写+非写分配 无延迟；回写 和读缺失一致
流水线执行的延迟受冒险和分层存储体系的影响，其中由分级存储体系(缺失)造成的额外空指令(泡泡)称为内存延迟周期(memory stall cycle)，减少缺失率和减少缺失惩罚是减少内存延迟的关键：
执行时间=[（执行的指令数目 × (CPI Avg + 内存延迟Avg）] × 时钟周期    内存延迟Avg = 缺失率Avg × 缺失惩罚Avg
利用空间局部性提高缓存性能(降低缺失率): 缓存“块”(cache line) —— 当内存单元发生缺失时将相邻内存引人缓存
利用空间局部性可以降低后面指令的潜在缺失，而块大小的选择会影响单指令的延迟(缺失时)和程序整体吞吐量之间的平衡: 块大小和缺失率的关系参考文中图例(有平衡点)

降低缺失惩罚
处理器和内存之间每次数据传输所需要的时间，称为总线周期时间。处理器和内存之间每个时钟周期传输的数据量称为内存带宽(内存吞吐量)。根据总线的位宽限制，内存系统可能需要多个总线周期来传递一个缓存块。但总线可以优先传输缺失的内存单元给CPU使得缺失延迟可以低于内存块传输周期。

# 计算机组成与设计——硬件软件接口
推测: 编译器或处理器推测指令结果以消除其他指令对该结果的依赖。在推测错误时能取消已执行指令的影响(恢复机制):
 - 软件 生成检查推测结果的指令并提供修复例程
 - 硬件 缓存推测结果直至检查后，取消缓存的操作结果并重新执行正确指令
推测会导致原本不存在的异常发生；推测在设计正确时能改善性能，而不慎使用可能降低性能！
指令级并行
 - 增加流水线深度以重叠更多指令
 - 增加多个功能单元以支持多发射(单时钟周期内可发射多条指令)：主要问题： 1、决定哪些指令被放到发射槽；2、处理数据和控制冒险
  - 静态多发射 由编译器在编译阶段实现，封装(寻找指令依赖关系并调度)多个指令成为发射包。超长指令字：可以同时启动多个操作的指令集，单个指令之间互相独立。
   - 静态双发射：指令1：ALU或分支 指令2：load或store 如果出现数据冒险使用nop替代(实际转变为单发射)
   - 循环展开： 循环体会被复制多份并且不同循环体中的指令可能会调度到一起以满足多发射(文中例子)，同时利用寄存器重命名(由寄存器重用导致的名字相关而非数据相关)提高吞吐。
  - 动态多发射 由cpu在执行阶段实现(超标量)，使用动态流水线调度(对指令重排序以避免阻塞)。
   - 动态流水线调度：取指译码单元 -> 调度器、保留站、重排缓冲区 -> 功能单元 -> 引退单元 可以把动态调度流水线想象为对程序数据流结构的分析过程。动态调度经常与基于硬件的推测机制相结合(分支预测)
为什么需要动态调度？ 1、不可预测的运行时(cache miss等)阻塞 2、不同硬件具有不同的流水线延迟和发射宽度 3、动态推测需要动态调度
通常静态和动态多发射需要互相协作，以掩盖阻塞的发生。虽然现代cpu支持多发射，但由于指令相关性和存储延迟，难以维持超过双射
